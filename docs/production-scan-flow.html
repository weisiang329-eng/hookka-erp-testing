1 · 核心概念 & 三种身份模式

8 个生产部门按身份追踪方式分 3 组：

身份模式 部门 追踪粒度 原因
BATCH FAB_CUT, FAB_SEW 按 Batch 追踪 (1 Batch = 1 FG) 布料 batch 色差约束，Batch 内 pieces 同 wipCode fungible，跨 Batch STRICT
POOL FOAM, WOOD_CUT, FRAMING, WEBBING 按 specKey fungible 池 无色差问题，同 spec 任意互换，FIFO by dueDate 自动分配
SO_BOUND UPHOLSTERY, PACKING 按 FGUnit (绑 SO) Upholstery Claim 时钉死 SO/customer，生成 FG 贴纸带客户资料

1.1 Fungibility 范围

  • BATCH 模式下：同 Batch + 同 wipCode → 互通 (例：Batch 001 的 2 个 Divan 互换)
  • BATCH 模式下：同 Batch + 不同 wipCode → 独立 (HB vs Divan)
  • BATCH 模式下：跨 Batch → STRICT (防布料 batch 串套)
  • POOL 模式下：同 specKey → 全互通（匿名池）
  • SO_BOUND 模式下：每个 FGUnit 唯一身份（绑 SO），不 fungible

2 · 数据模型

2.1 Batch（FAB_CUT / FAB_SEW）

Batch {
  id, batchNo (001/002/...), batchCode ("BD-20260420-001")
  productClass: "BD" | "SF"
  productCode, sizeCode, fabricCode
  dueDate, createdAt
  pieces: BatchPiece[]
  sourcePoIds: string[]
  // 分配信息（Upholstery Claim 时填）
  assignedSoId?, assignedSoNo?, assignedPoId?
  assignedCustomerName?, assignedAt?, assignedByWorkerId?
  fgUnitIds: string[]
  status: OPEN | READY_FOR_UPHOLSTERY | CLAIMED | UPHOLSTERED | PACKED | CANCELLED
}

BatchPiece {
  pieceCode ("BD DV-20260420-001"), pieceType ("HB" | "DV" | ...)
  wipCode ('8" Divan- 5FT'), count (2 for Divan, 1 for HB)
  specKey (fungibility key)
  slots: {
    deptDone: { FAB_CUT: number, FAB_SEW: number }
    slotPics: Array[count] of {pic1Id?, pic2Id?, dept}[]
    qrShortCodes: string[count]
    history: BatchScanEvent[]   // 追溯性
  }
}

BatchScanEvent {
  timestamp, deptCode, slotIndex, workerId, workerName
  picSlot: 1 | 2, action: "COMPLETE" | "UNDO"
}

2.2 SpecPoolSlot（FOAM / WOOD_CUT / FRAMING / WEBBING）

SpecPoolSlot {
  id, specKey, deptCode
  poId, jobCardId, wipCode, dueDate
  status: PENDING | DONE | CONSUMED  // CONSUMED = 被 Upholstery Claim 扣减
  doneAt?, doneByWorkerId?, doneByWorkerName?
  qrShortCode
  consumedAt?, consumedBy?  // 哪次 Upholstery Claim 扣的
}

2.3 FGUnit（Upholstery Claim 时生成）

FGUnit {
  id, unitSerial ("SO-2604-173-01-R1-U01-P1/3"), shortCode ("711993-1")
  soId, soNo, soLineNo, poId, poNo
  batchId, batchCode            // 追溯到 batch
  productCode, productName
  unitNo, totalUnits, pieceNo, totalPieces, pieceName
  customerName, customerHub
  mfdDate

  status: CLAIMED | UPHOLSTERED | PACKED | LOADED | DELIVERED | RETURNED

  claimedAt, claimedByWorkerId, claimedByWorkerName
  upholsteryCompletedAt, upholsteryPic1Id, upholsteryPic2Id, ...
  packedAt, packerId, packerName
  loadedAt, loadedOntoDoId
  deliveredAt, deliveredToSignature?, deliveredConfirmBy?
  returnedAt, returnReason?

  doId?                         // 挂哪张 DO
  scanHistory: FGScanEvent[]    // 全链路扫码追溯
}

FGScanEvent {
  timestamp, action, deptCode?, workerId, workerName
  fromStatus, toStatus, notes?
}

2.4 DeliveryOrder（加 Master QR）

DeliveryOrder {
  id, doNo ("DO-2604-0012")
  doQrCode ("DO|do-abc123")    // Master QR，客户一扫即签收全部
  customerId, hubId, customerName, hubShortName
  fgUnitIds: string[]          // 挂的 FG
  status: PENDING_DISPATCH | IN_TRANSIT | PARTIALLY_DELIVERED | DELIVERED | CANCELLED
  createdAt, createdBy
  dispatchedAt?, dispatchedBy?, truckNo?
  deliveredAt?, deliveredConfirmBy?
  remarks?
}

5 · 完整流程（10 Step）

用例：The Conts 下 3 张同 spec 的 1013 King 单 (SO-2604-173-01/02/03)。每张 1 set = 1 HB + 2 Divan。总共 3 HB + 6 Divan。

Step 0 · 下单 (Sales)

Sales 录入 3 张 SO，系统自动开 3 张 PO。

系统同时在 Spec Pool 预生成 slots：

  • WOOD_CUT pool ← 3 HB 木架 + 6 Divan 木框 slots
  • FRAMING pool ← 3 HB + 6 Divan slots
  • WEBBING pool ← 6 Divan slots
  • FOAM pool ← 3 HB + 6 Divan slots

所有 slots 带 dueDate 继承自 PO，FIFO 排序用。此时还不生成 FGUnit。

Step 1 · FAB_CUT (Batch 开 + 裁布)

Ali (裁布师傅)

1.1 Generate Batches：Ali 点 Generate Batches → 系统按 PO qty-set 生成 Batch（1 set = 1 Batch），打 9 张 Batch 贴纸 (50×75mm)。

1.2 裁布：同 spec 的布一起切。每片贴对应的 Batch 贴纸（订上去）。

1.3 Scan：Scan 贴纸 QR → batchPiece.deptDone.FAB_CUT += 1。记 slot PIC。

Ali scan BATCH|batch-001|DV|1
  → batch 001 DV piece deptDone.FAB_CUT: 0 → 1 (1/2), slotPics[0] = Ali

Ali scan BATCH|batch-001|DV|2
  → deptDone.FAB_CUT: 1 → 2 (2/2), slotPics[1] = Ali
  → batch 001 DV piece FAB_CUT 完成

Step 2 · FAB_SEW (按 Batch 缝)

Siti & Fatimah (缝纫师傅)

布料按 Batch 堆放。工人拿一片片缝。Fungibility：

✅ Batch 内 Divan 互换：同 Batch 的 2 个 Divan 互换，谁先缝好都算。
⛔ 跨 Batch STRICT：Batch 003 的 Divan 不能 mark 到 Batch 001 的 slot。

每 batch 所有 pieces 的 FAB_SEW 完成 → batch.status: OPEN → READY_FOR_UPHOLSTERY。

Step 3 · FOAM / WOOD_CUT / FRAMING / WEBBING (Spec Pool 并行)

4 个部门独立并行跑，不按 batch 分组：

  • WOOD_CUT 最早可开工（不依赖布料）
  • FRAMING 依赖 WOOD_CUT 出材料
  • WEBBING 依赖 FRAMING 出框架
  • FOAM 依赖 FAB_SEW 出布套（海绵按布 spec 做）

工人 scan Pool QR → poolSlot.status: PENDING → DONE，记 doneBy。

池子 slot 归属规则：不预先绑 PO。系统维护"该 spec 下总共需要 N 个，已完成 M 个"。Upholstery Claim 时再逻辑上挑出所需数量。

Step 4 · UPHOLSTERY Queue — ⭐ SO 分配时刻

Kumar (Upholstery 师傅)

4.1 打开 Queue 页面：FIFO by dueDate 列出所有可 Claim 的 SO。

4.2 Pre-req 检查：每个 SO 显示前置是否备齐：

  • Batch ready (FAB_SEW complete)
  • FOAM pool 够数（by specKey）
  • WOOD_CUT / FRAMING / WEBBING pools 够数

不满足 → Claim 按钮 disable，tooltip 说明缺什么。

4.3 点 Claim，系统做：

  1. 从 readyForUpholsteryBatches() 挑一个 batch（任意 ready 的都可）
  2. 从 Spec Pool 扣减所需 slots (FOAM 3 + WOOD 3 + FRAMING 3 + WEBBING 2)
  3. Batch 绑 SO：batch.assignedSoId 等
  4. 生成 FGUnit（每个 piece 1 条，共 3 条：HB + 2 Divan）
  5. 打印 FG 贴纸 × 3 (100×150mm)

4.4 领料 + 贴标：Kumar 拿着 FG 贴纸去仓库，从 Batch 堆和 Pools 里拿料，订 FG 贴纸到每件半成品上（老 Batch 贴纸可保可撕）。

Step 5 · UPHOLSTERY Complete

Kumar 做完一件 → scan 订着的 FG QR。

Kumar scan FG|fgu-01-HB
  → FGUnit.status: CLAIMED → UPHOLSTERED
  → upholsteryPic1Id = Kumar, upholsteryCompletedAt = now

3 件全部 UPHOLSTERED → 推到 Packing 工位。

Step 6 · PACKING

Yusof (Packing 师傅)
  1. 取下订着的 FG 贴纸（纸暂在手）
  2. 把半成品装进纸箱
  3. Scan 手里的 FG QR
  4. 把 FG 贴纸贴到箱子外面
  5. 推到待发货区
Yusof scan FG|fgu-01-HB
  → 检查: status 必须 = UPHOLSTERED ✅
  → 若 status 是 CLAIMED → ⛔ "Not finished upholstery yet"
  → status: UPHOLSTERED → PACKED
  → packedAt = now, packerId = Yusof

Step 7 · Pending Delivery (逻辑视图)

定义：FGUnit.status = PACKED AND doId IS NULL。

Logistics / Sales 在该列表页按 customer / SO / due date 多选，点 Create DO 建单。

Step 8 · Create DO + 打印 DO 单据

POST /api/delivery-orders → 系统生成：

  • 新 doNo（如 DO-2604-0012）
  • 新 doQrCode Master QR（如 DO|do-abc123）
  • 把选中的 FGs 的 doId 全部绑到这张 DO
  • DO status = PENDING_DISPATCH

打印 A4 DO 单据（含 Master QR + items 明细 + signature 栏）。

Step 9 · LOADING (装货)

装货工人 scan FG 贴纸。系统校验：

  • FGUnit.status = PACKED ✅
  • FGUnit.doId 非空 ✅
  • DO.status = PENDING_DISPATCH ✅

推进：FGUnit.status: PACKED → LOADED。当 DO 内所有 FG 都 LOADED → DO.status: PENDING_DISPATCH → IN_TRANSIT。

⛔ 没挂 DO 的 PACKED 被扫 → "This FG has no DO yet. Please create DO first."

Step 10 · DELIVERY (Master QR 一键签收)

司机送到客户 Hub，客户 / 司机 scan DO Master QR：

Scan DO|do-abc123
  → 检查 DO.status = IN_TRANSIT ✅
  → 弹窗 "Confirm delivery of all N items?"
      [ ✅ Yes, all received ]
      [ ⚠ Partial delivery — specify short items ]

Yes → 批量更新：
  DO.status = DELIVERED
  所有 FGUnit.status = DELIVERED, deliveredAt = now
  触发 SO 完成判定
  触发 Invoice 流程

部分签收：选 Partial → 进勾选界面，未到的标 SHORT_SHIP + 生成异常单，DO.status = PARTIALLY_DELIVERED。

7 · 扫码防呆矩阵

统一 POST /api/scan 端点。Body: { qrRaw, workerId }。系统按 QR 前缀识别类型 + 状态自动决定动作，前端只弹出确认窗。

QR 类型 当前状态 弹窗动作 结果 / 防呆
BATCH slot 未完成 Complete this piece @ {dept} deptDone +1, 记 PIC, slot full → PIC_FULL 拒绝
BATCH slot 已完成 ⛔ Already complete @ {dept} 显示历史 PIC，禁止操作
POOL PENDING Complete @ {dept} slot DONE, 记 doneBy
POOL DONE / CONSUMED ⛔ Already consumed 显示 consumedBy (哪次 Claim 扣的)
FG CLAIMED Upholstery Complete 选 PIC worker (max 2) → UPHOLSTERED
FG UPHOLSTERED Pack 选 packer → PACKED
FG PACKED, no DO ⛔ Wait for DO creation 提示 logistics 先建 DO
FG PACKED + DO=PENDING_DISPATCH Load & Dispatch → LOADED, 全部 LOADED → DO IN_TRANSIT
FG LOADED ⛔ Already loaded, use DO QR 指引用 DO Master QR
DO PENDING_DISPATCH ⛔ Not dispatched yet 需先 scan FGs load
DO IN_TRANSIT ✅ Confirm Delivery (all FGs) [Yes, all] / [Partial] 两路径
DO DELIVERED ⛔ Already received 显示 deliveredAt + 签收人
通用防呆：
  • 同一 worker 同一 slot 不能占 2 个 PIC 槽 (ALREADY_PIC)
  • 3 秒 debounce：避免双击重复 scan
  • 跨级状态推进拒绝：必须按顺序 CLAIMED → UPHOLSTERED → PACKED → LOADED → DELIVERED
  • Supervisor 可通过 /supervisor/scan override，但需填原因

8.2 Return 触发

QMS module 的 Returns tab 已有"Scan QR to create case"功能。新流程：

  1. Scan FG QR → 系统识别 DELIVERED 状态 → 弹 "Create Return Case?"
  2. 输入原因 (布料色差 / 结构损坏 / 尺寸错误 / ...)
  3. FGUnit.status: DELIVERED → RETURNED, returnedAt, returnReason
  4. 自动生成 Return Case，关联 FG + 全部生产历史
  5. 路由到 QC / 生产主管 review

8.3 Rework 触发

Rework 可在任一阶段触发（Upholstery 后发现 bug 也可）：

  1. Scan FG QR → supervisor 模式 → 选 Rework
  2. 选 Rework dept（哪个部门要重做）
  3. 系统把 FGUnit 状态回滚到该 dept 前一状态
  4. 加 rework note 到 scanHistory
  5. 该部门工作队列多出一项 "REWORK: {serial}"
重要：Rework 不生成新 FGUnit，贴纸不换，延续原身份。追溯链不断。

9 · 异常处理

9.1 Batch Release (Upholstery Claim 后反悔)

场景：Kumar 领料后发现布有瑕疵，不做了。

POST /api/batches/:id/release
  → batch.assignedSoId = null
  → batch.status = OPEN
  → 回退 Spec Pool 扣减
  → 删除已生成但未 UPHOLSTERED 的 FGUnit
  → 该 batch 回到 Upholstery Queue 可重新 Claim

9.2 DO Cancel

场景：DO 建了但客户改单，取消。

POST /api/delivery-orders/:id/cancel
  → DO.status = CANCELLED
  → 回退：挂的 FGs 的 doId 清空
  → FG 状态保持 PACKED (回到 Pending Delivery bucket)

9.3 Short Ship (Partial Delivery)

扫 DO Master QR 时选 Partial → 勾选界面：

  • Received → FGUnit → DELIVERED
  • Short → FGUnit 保持 LOADED，生成 SHORT_SHIP exception ticket
  • DO.status → PARTIALLY_DELIVERED
  • Short FG 进 "待重派" 列表，可重新加到新 DO

10.1 统一 /api/scan 分派逻辑

POST /api/scan
Body: { qrRaw: string, workerId: string, extras?: {...} }

1. Parse qrRaw → 按前缀识别 BATCH / POOL / FG / DO
2. 根据类型 + 当前状态决定 action:
   - BATCH   → recordBatchScan(...)
   - POOL    → recordPoolScan(...)
   - FG      → 按 FGUnit.status 推进状态机
   - DO      → 按 DO.status 决定 (IN_TRANSIT → deliver)
3. 返回:
   {
     success: boolean,
     action: "BATCH_COMPLETE" | "UPHOLSTERY_COMPLETE" | "PACK" | "LOAD" | "DELIVER_ALL" | ...
     nextPrompt: { fields: [...], confirmLabel: "..." }
     currentEntity: { batch? / fgUnit? / do? / poolSlot? }
     reason?: string,  // 拒绝时的 code
   }
打印成 PDF： 在浏览器打开此 HTML → Ctrl+P (或 Cmd+P) → 目的地选 "另存为 PDF" → 保存即可。纸张选 A4，页边距 "默认"。
